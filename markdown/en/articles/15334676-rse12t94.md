# Folders and documents access control v2

How access levels work for documents and folders, plus a quick-reference for the access level each operation requires. When you select one or more documents and/or folders in the documents section, the access control menu appears in the right information menu. There you can see who has access, and edit it.

### **How to edit access**

1. Select the document(s) and/or folder(s) you want to configure.
1. Open the right information menu.
1. Click **Edit access**.

> **Important:** **Access required:** Full access

## 1. **1. Access levels (4 levels)**

There are four access levels. The level that applies is the **highest** one a member has received, either through their user access or through any of their teams (exceptions below).

## 2. **2. Which access level applies (3 steps)**

The access with the highest weight always wins, but there are two exceptions. Restrictive tiers like "No Access" will be elevated if any other pathway grants higher rights. Evaluate in this order to be certain:

1. **Is the member an administrator?**
→ **Always full access (fixed)**. Administrators can do everything in the project.
1. **Does the member have a single-user setting?**
→ **That exact level applies**. This is the most specific setting and overwrites all others — whether higher or lower.
1. **Otherwise**
→ **The highest access applies**: the maximum of "all users," "team" (the highest one if they belong to several), and "owner."

> **Important:** Once you internalise these three steps, every individual comparison in the old article (all users vs team, owner vs team, and so on) reduces to: "it's the maximum — only single-user and administrator are exceptions."Owner = the person who uploaded the file or created the folder. Owners have full access to their own content by default, for data-privacy reasons.

### 2.1 **Recommended workflow**

Set access **per team rather than per user**. Roles change often, and team-based access stays flexible: a member added to a team gets the right access as soon as they join the project. A common pattern is to set "all users" to no access — so new, not-yet-assigned members can't see sensitive information — then grant access to each team as needed.

## 5. **5. Overwrite options (scope applied on save)**

When you save the access dialogue, the access on the selected elements is **overwritten regardless of what was previously configured**. The three options control how far that change reaches.

<div class="intercom-interblocks-table-container"><table role="presentation"><tbody><tr><td style="background-color: #e3e7fa80;"><p><b>Option</b></p></td><td style="background-color: #e3e7fa80;"><p><b>What gets overwritten</b></p></td><td style="background-color: #e3e7fa80;"><p><b>When to use</b></p></td></tr><tr><td><p>Folder and new content</p></td><td><p>Selected elements + any new elements created in them</p></td><td><p>When you must not change the access of existing contents</p></td></tr><tr><td style="background-color: #e8e8e880;"><p>Folder and files<br/>(default)</p></td><td style="background-color: #e8e8e880;"><p>The above + existing documents one level down</p></td><td style="background-color: #e8e8e880;"><p>The usual choice; per-subfolder access is preserved</p></td></tr><tr><td><p>Folder and all subfolders and files</p></td><td><p>Everything below, all the way down the tree</p></td><td><p>Only when it's fine to overwrite existing subfolder access too</p></td></tr></tbody></table></div>

> **Note:** **Careful:** the first two options overwrite only the selected elements, so older access can remain on sub-elements. Members might no longer be able to navigate to them, yet still reach them through filtering.
